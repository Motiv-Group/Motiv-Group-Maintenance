## 1. Security & authorisation

The external audit found 3 real issues (X1–X3). Our deeper pass found 4 more (S1–S4), three of them worse than anything ChatGPT flagged. All 7 are **fixed on this branch**. Root cause was the same everywhere: the app's threat model treated everyone inside a company as trusted, but **suppliers are competing outsiders inside the same tenant** — and two pages predating v3 had no scoping at all.

- [x] **X1 — Supplier store page leaked all store tickets + competitors' quote amounts** (HIGH)
  `app/supplier/stores/[id]/page.tsx` fetched every ticket for the store with embedded `quotes(amount)`, gated only by same-company. Any supplier could read rival suppliers' quote amounts.
  **Fix (76d688e):** tickets filtered to awarded (`tickets.supplier_id`) or invited (`ticket_suppliers`) for the caller's `supplier_users` orgs — mirrors the ticket-detail gate; quotes embed dropped.
- [x] **X2 — Any supplier could reassign / wipe regional managers** (HIGH)
  `POST /api/supplier/assign-rm` let any supplier-role user upsert an RM onto a region, and its "clear" branch deleted **every** `regional_users` row for the region. Admin-era leftover; RM linking belongs to the system-admin Hierarchy tab only.
  **Fix (6c754ae):** route + `AssignRMForm` deleted; supplier store page stripped.
- [x] **X3 — Verification-doc reference check was `url.includes('supplier-docs')`** (MEDIUM)
  A crafted reference (any string containing `supplier-docs`) was stored and later **signed with the service-role client** on read-back. Exploitability low (paths contain user-UUID + timestamp + random), but real.
  **Fix (76d688e):** reference must parse (`bucketAndPath`) to bucket `supplier-docs` AND path prefix `${user.id}/`; `supplier-docs` added to `STORAGE_BUCKETS`.
- [x] **S1 — `/supplier/regional` listed every RM on the platform, ALL tenants** (HIGH)
  Admin-client query on `user_profiles` with `role='regional_manager'` and **no company filter** — full name/company/email/phone of every regional manager across every company, readable by any supplier including unverified self-signups. Orphaned page (not in nav).
  **Fix (89d0cb3):** page deleted.
- [x] **S2 — `/supplier/regional/[id]` — zero auth, any RM's contacts + full store estate** (HIGH)
  No `auth.getUser`, no company check — walk any user-id, get that RM's email/phone/address and every store (with physical addresses) in their regions.
  **Fix (89d0cb3):** page deleted.
- [x] **S3 — `GET /api/suppliers` returned every supplier row across all tenants** (HIGH)
  `select('*')` with no tenant filter (email, phone, VAT number, verification status). Dead route — its UI was deleted in `4a143c9` — but live and reachable. Siblings POST/bulk inserted rows with no `company_id` (MEDIUM).
  **Fix (89d0cb3):** all three `/api/suppliers*` routes deleted.
- [x] **S4 — Invited supplier could award themselves the ticket** (MEDIUM)
  `/api/tickets/[id]/transition` wrote `body.supplierId` into `tickets.supplier_id` for `require_assessment`/`request_quote` with no validation, and `hasAccess()` passes on a mere invite — an invited supplier could set `supplier_id` to themselves, flipping every "awarded" check (RM↔supplier chat, awarded-only surfaces) and pre-empting the competitive award.
  **Fix (89d0cb3):** `supplierId` is ignored for supplier-role callers; internal roles unaffected.

**Verified solid** (adversarially checked, no action needed): storage path confinement + atomic upload quota; WhatsApp webhook HMAC fail-closed; cron auth; supplier-scoped queries everywhere else (`.in('supplier_id', supplierIds)` pattern); `rmOwnsTicket()` on all RM mutations; quote-decision idempotency (no double-award); role escalation via `PATCH /api/profile` explicitly blocked; project routes fully tenant-scoped; admin routes re-check `system_admin` server-side; forgot-password doesn't leak account existence; zod validation on effectively all write routes; rate limits everywhere that matters.

**Still open:**
- [ ] Independent penetration test (owner-track; after these fixes, not before)
- [ ] E2E role tests that would have caught S1–S4 (see §4)
- [ ] `pdfjs-dist` worker loads from unpkg CDN at runtime (`components/admin/SendQuoteForm.tsx`) — external script dependency; consider self-hosting the worker (LOW)

**Score: 8.0** — the tenant/role model is now consistently enforced; what remains is external validation, not known holes.

## 6. Reliability & operations

**Good:** error boundaries exist; Sentry fully wired (server/edge/client + source maps), no-ops cleanly when DSN unset; rate limiter is production-grade (Upstash sliding window, in-memory fallback, Sentry alert on prod fallback, degrades open on Redis outage); cron auth solid.

- [x] **R1 — Error boundaries only `console.error`'d** — client render errors never reached Sentry (`onRequestError` covers server only). **Fixed:** both boundaries now `Sentry.captureException`.
- [x] **R2 — Notification purge never ran** — it lived only in the unscheduled `v3-recompute` route. **Fixed:** folded into the daily `v3-snapshots` cron (isolated try/catch); `v3-recompute` kept as a manual recompute-only trigger.
- [ ] **R3 — Zero timeouts/retries on external HTTP.** Groq transcription/extraction and WhatsApp Graph sends are bare `fetch` — a hung call rides to the function timeout and the WhatsApp user gets silence. Add `AbortSignal.timeout(…)` + one retry + Sentry capture on final failure.

- [ ] **R5 (owner) —** uptime monitoring + log-drain alerts; staging environment; incident runbook (all tracked in PATH_TO_9.5).
